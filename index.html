Scheduling jobs on a single machine

The Problem

In the single machine scheduling problem, a set of independent jobs must be processed on a single machine, where at most one job can be executed at any time. Each job Jj is associated with three parameters: a processing time, a release date, and a due date. The scheduler must determine the order in which jobs are processed so that a performance objective is optimized.

A job cannot begin before its release date — that is, the start time Sj ≥ rj. This makes the problem more realistic and more difficult than the classical case where all jobs are available at time zero.

pj   Processing time — amount of time required to complete job j
rj   Release date — earliest time at which job j becomes available
dj   Due date — preferred completion deadline for job j
Cj   Completion time — time when job j finishes
Fj   Flow time = Cj − rj
Lj   Lateness = Cj − dj
Lmax   Maximum lateness = maxj(Cj − dj)

Algorithm theory & approximations

FCFS — First Come, First Served

FCFS schedules jobs in the order of their release or arrival times. The earliest released available job receives service first. This method emphasizes fairness and simplicity rather than optimization quality.

Pseudocode

Input: Jobs with processing time pj and release time rj
t = 0
Unscheduled = all jobs

While jobs remain:
  Available = jobs with rj ≤ t
  If no job available:
    t = next smallest release date
  Else:
    Choose available job with smallest release time
    Process it completely
    t = t + pj
End

Explanation: The scheduler checks which jobs have already arrived. If the machine is idle and some jobs are available, it selects the one that arrived first. If no job has arrived yet, the machine waits until the next release date. FCFS is commonly implemented using a queue data structure.

Approximation Constant α

For most important objectives such as ΣCj, ΣFj, Lmax, FCFS does not possess a universal constant-factor approximation guarantee.

Reason: A very long early job can delay many short later jobs.

Example:

Job rj pj
J1 0 100
J2 1 1
J3 2 1

FCFS runs: J1 → J2 → J3, causing extremely poor total completion time.

Hence: α = Unbounded for many objectives.

SPT — Shortest Processing Time

SPT schedules the currently available job having the smallest processing time. It favors short jobs and is highly effective for reducing average waiting and completion time. By finishing short jobs first, it minimizes the total time all jobs spend waiting and being processed.

Pseudocode

Input: Jobs with processing time pj and release time rj
t = 0
Unscheduled = all jobs

While jobs remain:
  Available = jobs with rj ≤ t
  If no job available:
    t = next smallest release date
  Else:
    Choose available job with minimum pj
    Process it completely
    t = t + pj
End

Explanation: At every decision point, only released jobs are considered. Among them, the shortest job is selected. This helps many jobs complete early and reduces queue congestion.

Approximation Constant α

With Release Dates (1|rj|ΣCj):
Plain non-preemptive dynamic SPT is a practical heuristic, but no classical universal constant approximation ratio is assigned in standard literature. Therefore: α = No standard fixed bound

Important Exact Result (Without Release Dates):
If all jobs are available initially (rj = 0 ∀j), the problem becomes 1||ΣCj and SPT is optimal. Thus: α = 1

Proof (Pairwise Exchange):
Assume adjacent jobs i, j with pi > pj. Ordering i before j gives a larger total completion time than ordering j before i. So repeatedly swapping inversions produces ascending processing times. Hence SPT is optimal when all jobs are released at time zero.

EDD — Earliest Due Date

EDD schedules the currently available job whose due date is smallest. It prioritizes urgent jobs and is primarily used for lateness minimization.

Pseudocode

Input: Jobs with processing time pj, release time rj, due date dj
t = 0
Unscheduled = all jobs

While jobs remain:
  Available = jobs with rj ≤ t
  If no job available:
    t = next smallest release date
  Else:
    Choose available job with minimum dj
    Process it completely
    t = t + pj
End

Explanation: Whenever the machine becomes free, the scheduler considers all released jobs and selects the most urgent one (smallest due date). If no job is available, it waits. This is the dynamic version of EDD under release constraints.

Approximation Constant α

For the single machine scheduling problem with release dates and negative due dates (1|rj|Lmax), the EDD rule is a 2-approximation algorithm:

LmaxEDD ≤ 2 · Lmax*

Proof

Let job j be the job attaining maximum lateness in the EDD schedule:

LmaxEDD = Cj − dj

Look backward from Cj and find the earliest time t such that the machine was continuously busy during [t, Cj] with no idle gaps. Let S be the set of jobs processed in this interval. Since no job in S was available before t:

Cj = r(S) + p(S)

In any feasible schedule, all jobs in S must wait until r(S) and need total time p(S), so some job finishes no earlier than r(S) + p(S). Since all due dates are negative, d(S) < 0:

Lmax* ≥ r(S) + p(S) − d(S) ≥ r(S) + p(S) = Cj

Separately, even in the optimal schedule job j cannot start before rj and requires pj time. Since rj, pj ≥ 0:

Lmax* ≥ rj + pj − dj ≥ −dj

Adding the two lower bounds:

2 · Lmax* ≥ Cj + (−dj) = Cj − dj

2 · Lmax* ≥ Cj − dj = LmaxEDD

∴   LmaxEDD ≤ 2 · Lmax*

Therefore EDD is a 2-approximation algorithm for 1|rj|Lmax under negative due dates.

Special Case: All jobs available initially

If all jobs are available at time zero (rj = 0 ∀j), then the problem reduces to (1||Lmax). In this case, the EDD rule produces an optimal schedule.

LmaxEDD = Lmax*

Hence, the approximation constant becomes:

α = 1

Thus, without release times, EDD is not just an approximation algorithm but an optimal algorithm.
